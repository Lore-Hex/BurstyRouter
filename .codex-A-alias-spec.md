## 1. Model aliasing (`-alias from=to`, repeatable; env `BURSTY_ALIASES=a=b,c=d`)
The direction is CLOUD-id → LOCAL-model: users configure e.g.
`-alias gpt-4o=qwen2.5-coder:32b -alias anthropic/claude-haiku-4.5=llama3.2`.
- config: repeatable flag (flag.Var with a map value) + comma-separated env;
  validate `from=to` shape, reject dup keys.
- policy.Decide: when the inbound model matches an alias key (exact match,
  after handling the `local/` prefix which still wins), the LOCAL forward
  rewrites the model to the alias TARGET (reuse ReplaceTopLevelString in the
  LocalBody path) while TRBody KEEPS the original inbound model — bursts must
  send the cloud id the user asked for.
- An aliased request is local-CAPABLE under default policy (not forced): full
  → burst to TR with the ORIGINAL model id; error → same. Explicit pins
  (provider.only local / local/ prefix) behave as today.
- handleModels merged list: each alias key appears as a model entry
  (owned_by "bursty-alias", metadata noting the local target) so tool
  dropdowns (Cursor) list the ids users configured.

## 2. Burst-model correctness fix (real bug)
Today a default-policy request for a LOCAL-native id (e.g. "llama3.2")
that bursts sends "llama3.2" to TrustedRouter, which cannot serve it — the
burst fails in the worst moment. Fix policy/proxy:
- If the inbound model is NOT TR-namespaced (heuristic: no "/" and not a
  trustedrouter/* route) and no alias maps it, the request is effectively
  LOCAL-ONLY: on full/error return the local-only behavior (429 w/
  Retry-After on full; surface the local error on error) instead of a doomed
  TR burst. Add `-burst-fallback-model` (env BURSTY_BURST_FALLBACK_MODEL,
  default empty): when set, such requests DO burst with that model id
  substituted into TRBody (and the response's model field left as TR returns
  it). Document both behaviors.
- Stats: new counter bursts_skipped_unmapped.

## 4. x-api-key inbound auth (real bug)
authorized() only accepts `Authorization: Bearer <token>`; Anthropic-family
clients (Claude Code with ANTHROPIC_API_KEY) send `x-api-key: <token>` — so
BURSTY_TOKEN breaks them today. Accept EITHER header (constant-time compare
both); tests for both headers on /v1/* incl. /v1/messages, and /healthz
stays open.
